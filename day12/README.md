## 2. 为什么需要输出解析

大模型默认的输出通常是自然语言。

比如你问它：

```text
帮我总结这段文本的核心观点
```

它可能返回：

```text
这段文本主要讲了三个方面：第一是...
```

这很适合人看，但不一定适合程序处理。

如果你想把它接到后续系统里，比如：

- 自动填表
- 自动分流
- 自动生成数据库记录
- 自动生成工单

那就需要结构化输出。

### 2.1 自然语言输出的问题

自然语言虽然灵活，但有这些问题：

- 格式不稳定
- 容易多说废话
- 程序难解析
- 不适合自动化处理

### 2.2 结构化输出的优势

结构化输出可以让你拿到：

- 固定字段
- 明确类型
- 可校验内容
- 更容易接后续逻辑

比如：

```json
{
  "title": "Python 学习建议",
  "difficulty": "beginner",
  "tags": ["python", "programming", "beginner"]
}
```

程序看到这个结果后，就可以直接判断、存储、分发、统计。

## 3. Day 12 核心知识点

### 3.1 Output Parser

Output Parser 的作用是：

- 接收模型输出
- 按指定规则解析
- 转成 Python 对象、字典、JSON、Pydantic 模型等

### 3.2 结构化输出

结构化输出的本质是：

> 让模型按照你定义的字段和格式返回结果。

比如：

- 名称
- 年龄
- 邮箱
- 任务优先级
- 分类标签

### 3.3 JSON 输出

JSON 是最常见的结构化输出形式。

它的优点是：

- 通用
- 可读
- 易解析
- 前后端都能用

### 3.4 错误处理

模型输出并不总是完美的，常见问题包括：

- 少括号
- 多解释文字
- 字段名写错
- 值类型不对
- JSON 格式不合法

所以你必须学会：

- 检查输出
- 捕捉异常
- 尝试修复
- 必要时重试

## 6. 本日建议的实现流程

### 第 1 步：定义输出目标

先想清楚你要模型输出什么。

例如：

- 用户意图
- 任务分类
- 情绪判断
- 关键词提取
- 邮箱地址
- 简历信息

你越清楚字段，后面越容易解析。

---

### 第 2 步：设计 schema

schema 就是输出规则。

例如你希望模型输出：

```json
{
  "name": "张三",
  "age": 18,
  "hobbies": ["阅读", "编程"]
}
```

那你就要提前规定：

- `name` 是字符串
- `age` 是整数
- `hobbies` 是数组

---

### 第 3 步：写提示词

提示词里要明确要求模型：

- 只能输出 JSON
- 不要输出多余解释
- 字段必须完整
- 字段名必须固定

比如：

```text
请严格按照 JSON 格式输出，不要添加任何额外解释文字。
```

---

### 第 4 步：解析输出

拿到模型输出后，做 JSON 解析。

如果解析成功，就得到结构化对象。

如果解析失败，就进入错误处理流程。

---

### 第 5 步：校验和修复

解析成功不等于内容正确。

你还要检查：

- 字段是否缺失
- 字段类型是否正确
- 是否包含非法内容

如果有问题，可以：

- 重新提示模型修正
- 给默认值
- 返回错误信息

## 10. 常见错误

### 10.1 模型输出多余解释

比如它说：

```text
好的，下面是 JSON：
{...}
```

这会影响程序解析。

#### 解决方式

- 强调“只输出 JSON”
- 用更严格的 prompt
- 解析前先提取 JSON 主体

---

### 10.2 字段名不一致

比如你要的是 `intent`，模型输出了 `task_type`。

#### 解决方式

- schema 统一
- 提示词明确
- 解析后统一映射

---

### 10.3 类型错误

比如：

- 你要整数，模型给了字符串
- 你要数组，模型给了单个字符串

#### 解决方式

- 解析后做类型校验
- 不符合就重试

---

### 10.4 JSON 格式非法

比如少了引号、少了逗号、括号不配对。

#### 解决方式

- 捕捉 `json.JSONDecodeError`
- 重新提示模型严格按 JSON 输出
